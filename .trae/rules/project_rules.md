## 二、多对话协作模式（新增）

本项目采用**多对话流水线**协作，核心约束如下：

### 角色与权限

| 对话角色 | 职责 | 禁止行为 |
|---|---|---|
| **计划/审查对话** | 定任务、写任务卡、审查代码、输出审查结论、推送部署 | **不直接改代码** |
| **执行对话** | 按任务卡开发、实现功能、提交代码 | 不自行变更需求范围 |
| **审查对话** | 对执行对话产出做二次审查（可选） | — |
| **修 Bug 对话** | 处理测试失败、线上问题 | — |

### 关键约束

- **串行流水线**：同一时间只有一个对话在改代码
- **契约优先**：开发前必须有明确的任务卡（验收标准 + 检查点）
- **审查闭环**：执行完成后必须回到计划/审查对话验收，不通过则输出修复任务卡
- **当前任务卡单一来源**：当前任务只写入 `CURRENT_TASK.md`，执行对话以该文件为准

### 标准流程

```
1. 用户提出需求 → 计划对话调用 Skill 制定方案
2. 输出任务卡（目标、范围、验收标准、验证命令）
3. 执行对话按任务卡开发
4. 执行完成后 → 计划对话审查（构建/代码/业务/标准）
5. 发现问题 → 输出修复任务卡
6. 执行对话修复 → 再次审查
7. 通过后 → 计划对话推送代码、触发部署
8. 线上验证 → 关闭本轮迭代
```

### 任务卡模板

```markdown
## 任务：[任务名称]

### 目标
[一句话描述]

### 范围
- [ ] [具体项1]
- [ ] [具体项2]

### 验收标准
- [ ] 构建通过：`pnpm build`
- [ ] 无固定色残留：`rg "bg-white|text-zinc" src/app/admin`
- [ ] 业务逻辑不变：认证/权限/发布流程未破坏
- [ ] 视觉达标：暗夜模式无白块、无低对比文字

### 验证命令
```bash
pnpm build
```

### 禁止
- 不改 API、数据库、权限、发布逻辑
- 不重构未要求的内容
```

### 审查清单

| 检查项 | 标准 |
|---|---|
| 构建 | `pnpm build` 通过 |
| 固定色 | 后台无 `bg-white`、`text-zinc-*`、`bg-zinc-*` 残留 |
| 业务逻辑 | 认证、权限、发布状态、数据隔离未破坏 |
| 主题适配 | 暗夜模式无大面积白块、无低对比文字 |
| 代码风格 | 匹配项目现有风格，无新增注释（除非用户要求） |

### 当前任务卡轮换规则

| 状态 | 操作 |
|---|---|
| 执行前 | `CURRENT_TASK.md` 只保留当前一张任务卡 |
| 审查通过 | 清空当前任务卡，写入下一张任务卡 |
| 审查不通过 | 清空当前任务卡，写入修复任务卡 |
| 全部完成 | 清空任务卡，写入“当前无任务” |

## 三、编码原则（Karpathy四大原则）

### 1. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

- 不确定就说出来，不要静默假设
- 有多种理解时，列出来让决策者选
- 有更简单的方案时，说出来
- 不清楚就停下来问

### 2. Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- 不做没要求的功能
- 不为单次使用的代码做抽象
- 不做没要求的"灵活性"和"可配置性"
- 200行能写成50行，就重写

### 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

- 不"顺手"改相邻代码、注释、格式
- 不重构没坏的东西
- 匹配已有风格，即使你不会这样写
- 你改的代码产生的废弃引用/变量，你负责删；之前就存在的废弃代码，只提不改

### 4. Goal-Driven Execution

**Define success criteria. Loop until verified.**

- "加验证" → "写失败测试，然后让它通过"
- "修Bug" → "写复现测试，然后让它通过"
- "重构X" → "确保重构前后测试都通过"

多步任务先列计划：
```
1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
3. [步骤] → 验证: [检查点]
```

## 四、协作流程

| 步骤 | 动作 | 负责人 |
|------|------|--------|
| **1. 定任务** | 写清楚「什么叫完成」+ 验收标准 | 基石+门神 |
| **2. 定契约** | 接口契约（OpenAPI/设计稿）确认 | 基石+前后端 |
| **3. 做** | 按契约开发 | 玄武/绘魂/... |
| **4. 验收** | 按验收标准查，含契约一致性 | 门神+基石 |

## 五、验收标准模板

每个任务必须有验收标准，具体指标以项目技术规范为准。

```markdown
## 验收标准

### 1. 文档
- [ ] 有完整方案
- [ ] 有接口契约（后端）/ 设计稿（前端）

### 2. 代码
- [ ] 代码已提交
- [ ] 符合项目技术规范
- [ ] 错误码使用与项目统一错误码体系一致

### 3. 测试
- [ ] 测试覆盖率达标（以项目技术规范为准）
- [ ] 所有测试通过

### 4. 运行
- [ ] 能运行
- [ ] 核心流程通

### 5. 业务规则（涉及核心业务逻辑时）
- [ ] 数值公式与产品需求一致
- [ ] 边界条件已覆盖
```

## 六、需求变更

发现需求与实现矛盾时：

1. **不自行决定**，先提出问题
2. 基石评估影响 → 产品确认 → 文档更新 → 再开发
3. 紧急情况：基石直接决策，事后补确认

|--------|------|------|
