# AI之翼 完整产品设计文档

> 主题：AI 场景式学习与能力进阶网站
> 产品目标：交付一个成熟可运营的网站，包含前台学习体验、管理后台、内容管理与发布能力；后续新增场景时只需要补充内容，不需要重新设计或编程网站本身。

## 1. 产品一句话定义

AI之翼是一个按“我想学会什么场景能力”组织的 AI 学习与提升路径网站，帮助普通人在真实场景中沿统一四阶段路径，逐步掌握可迁移、可复用、可稳定复现的 AI 能力。

## 2. 完整产品目标

AI之翼要交付的是一个成熟网站，而不是临时验证页、说明书式样板或静态内容样板。

| 目标 | 定义 |
|---|---|
| 前台成熟 | 用户能从首页进入场景，阅读完整学习路径，进入学习节点，按练习完成能力证据，并继续下一阶段 |
| 后台成熟 | 管理员能在后台维护场景、阶段、学习节点、工具建议、案例内容、发布状态和排序 |
| 内容可扩展 | 后续新增场景时，只补内容与配置，不需要改页面结构、组件逻辑或路由设计 |
| 结构可复用 | 所有场景共用统一内容模型、统一页面模板、统一节点模板、统一管理后台表单 |
| 运营可持续 | 网站具备内容草稿、预览、发布、下线、排序、SEO、基础数据查看等运营必需能力 |

完整产品必须回答的是：

1. 用户想学一个 AI 场景时，能否快速找到入口
2. 用户进入场景后，能否看到完整、真实、可练习的学习路径
3. 用户是否知道每一步该如何和 AI 对话、如何补充信息、如何检查结果
4. 管理员是否能不写代码新增和维护场景内容
5. 后续扩充更多场景时，网站是否不需要重新设计和编程

## 3. 品牌定位

| 维度 | 定义 |
|---|---|
| 品牌名 | AI之翼 |
| 品牌使命 | 给每个人装上 AI 的翅膀，让能力增长变成看得见、练得成、能迁移的过程 |
| 品牌主张 | 不是泛学 AI，不是堆工具，而是围绕真实目标，逐步学会、稳定掌握并持续升级 AI 能力 |
| 核心心智 | 用户不是来找工具，也不是来找代做，而是来找到一条场景化学习路径，学会以后能稳定做到 |
| 产品形态 | 前台学习网站 + 管理后台 + 内容管理系统 + 可扩展场景库 |

## 4. 核心用户与后台角色

### 4.1 前台用户

| 用户状态 | 典型问题 | 产品回应 |
|---|---|---|
| 有目标，但不知道怎么开始 | 我想学会用 AI 做成一类事情，但不知道第一步是什么 | 通过场景入口和四阶段路径给出学习起点 |
| 做出来过，但不稳定 | 为什么我偶尔能做出来，但换个输入就不稳了 | 用能力标准、练习任务和能力证据帮助复现 |
| 已经在用一些工具 | 现在的工具够不够，什么时候该继续深挖，什么时候该迁移 | 在每个阶段后置给出当前工具和更优工具建议 |
| 想继续提升 | 下一阶段该学什么，怎样才算真的升级了 | 通过持续进阶阶段和升级练习引导优化 |

### 4.2 后台角色

| 角色 | 权限 |
|---|---|
| 管理员 | 管理用户、全站配置、所有内容、发布状态、SEO、排序、数据查看 |
| 内容编辑 | 创建和编辑场景、阶段、节点、案例、工具建议，提交预览或发布审核 |
| 审核发布者 | 检查内容质量、发布、下线、调整推荐位 |

### 5.3 不做项

| 不做项 | 原因 |
|---|---|
| 代替用户执行账号运营 | 产品目标是教会用户，不是代做服务 |
| 工具排行榜 | 工具服务学习阶段，不做独立工具导航站 |
| 社区复杂体系 | 后续可扩展，但不是成熟内容学习站的必要首交付 |
| 自动替用户判断所有答案对错 | 能力证据以用户自查和结构化提交为主，可后续扩展智能评估 |

## 7. 统一四阶段学习骨架

| 阶段 | 定义 | 用户感受 | 能力标准 |
|---|---|---|---|
| 理解 | 了解这个场景要解决什么问题、适用边界在哪里 | 我看懂了 | 能清楚说明目标、输入输出和适用边界 |
| 上手 | 跟着明确步骤完成第一次最小练习 | 我能跟着做了 | 能在辅助下完成最小练习，并说明每一步为什么存在 |
| 独立稳定产出 | 脱离逐步跟做，独立完成同类练习，并在变化输入下稳定复现 | 我会用了 | 能独立完成一次，并在变化输入下再完成一次 |
| 持续进阶 | 能比较方法、优化流程、迁移到更复杂或相邻场景 | 我在升级 | 能说明优化前后差异，并完成一次升级练习 |

## 8. 阶段能力标准与能力证据

| 阶段 | 能力证据 | 判断标准 | 常见未达标原因 | 补强动作 |
|---|---|---|---|---|
| 理解 | 一段自己的场景说明 | 能说明目标、输入输出、边界，并举出一个适合 / 不适合自己的例子 | 只会复述原文，不会结合自己场景 | 重写成自己的任务描述 |
| 上手 | 一次最小练习结果 + 步骤解释 | 练习真实完成，且能解释每一步为什么存在 | 照抄步骤但说不清原因 | 补充步骤解释，重新演示 |
| 独立稳定产出 | 第二次变化输入下的结果 + 自查记录 | 能在输入变化后再完成一次，并记录修正过程 | 只能重复原案例，无法迁移 | 更换输入再做一轮，并按清单自查 |
| 持续进阶 | 优化前后对比结果 + 复盘 | 能说明效率、质量、稳定性或复杂度中至少一项提升 | 只有主观感受，没有前后证据 | 补交前后对比与复盘 |

## 9. 重点场景样板：用 AI 运营一个公众号

重点不是展示 AI 能写文章，而是让用户学会在账号定位、选题、写作、发布、分发、复盘和迭代中具体使用 AI。

| 阶段 | 能力标准 | AI 介入方式 | 工具建议口径 |
|---|---|---|---|
| 理解 | 说清楚公众号服务谁、解决什么问题、栏目如何持续、发布后看哪些指标 | 用 AI 追问账号定位、读者问题、栏目边界和复盘指标，输出一份可检查的运营链路说明 | 当前常用 AI 聊天工具即可，先不迁移 |
| 上手 | 完成第一篇文章从选题卡、资料整理、大纲、正文、标题封面到发布前检查的完整准备 | 用 AI 分环节辅助选题校准、资料整理、大纲搭建、正文改写、标题生成和发布检查，不一次性要成稿 | 可继续用聊天工具；当素材和版本开始散乱时再建议长文编辑或素材管理工具 |
| 独立稳定产出 | 围绕同一账号定位连续完成三篇内容，并记录发布数据、复盘结论和下一篇调整 | 用 AI 复用账号设定、栏目标准和复盘表，整理数据与评论，但由用户决定下一篇调整 | 建议可保存上下文或项目资料的 AI 工具；内容增多后再迁移到内容日历或看板工具 |
| 持续进阶 | 基于真实数据选择一个瓶颈，升级选题、标题、结构、分发、互动或复盘机制 | 用 AI 分析三篇内容的共性问题，生成一个可执行升级方案，并对比升级前后变化 | 只有当内容数量、协作或数据复盘超出现有工具承载时，才建议更完整的内容运营工作台 |

## 10. 场景页结构

场景页必须以四阶段作为一级内容架构。场景定义和适用人群用于确认入口，进入正文后不再按“能力标准区、学习内容区、练习区、工具区”横向拆散，而是按“理解 / 上手 / 独立稳定产出 / 持续进阶”逐阶段展开。

| 一级区块 | 阶段内必须包含的内容 |
|---|---|
| 场景定义 | 这是什么问题，适合谁，不适合谁 |
| 四阶段总览 | 路径全貌，帮助用户理解接下来四个阶段 |
| 理解 | 能力标准、学习重点、最小练习、能力证据、AI 如何介入理解、当前工具怎么学和怎么用、更优工具何时需要 |
| 上手 | 能力标准、学习重点、最小练习、能力证据、AI 如何分环节辅助第一次产出、当前工具怎么学和怎么用、更优工具何时需要 |
| 独立稳定产出 | 能力标准、学习重点、最小练习、能力证据、AI 如何复用上下文并辅助复盘、当前工具怎么学和怎么用、更优工具何时需要 |
| 持续进阶 | 能力标准、学习重点、最小练习、能力证据、AI 如何发现瓶颈并辅助升级、当前工具怎么学和怎么用、更优工具何时需要 |
| 能力结果与下一步 | 走完后会掌握什么能力，以及从哪个学习节点开始 |

## 11. 学习节点页结构

| 区块 | 内容 |
|---|---|
| 当前学习目标 | 这一节点要掌握什么 |
| 前置条件 | 开始前要具备什么 |
| 为什么先学这个 | 它在整条路径中的作用 |
| 关键概念 | 必须理解什么 |
| 方法重点 | 要学会什么做法 |
| AI 对话示范 | 工具选择、提示词输入、追问轮次、用户补充、AI 输出、结果检查 |
| 练习任务 | 完成什么最小练习 |
| 当前工具怎么学、怎么用 | 保持已有习惯 |
| 更优工具怎么学、怎么用 | 后置给出迁移路径 |
| 常见错误 | 最容易卡住的点 |
| 通关标准 | 达到什么算完成 |
| 能力证据 | 需要交付什么 |
| 下一步 | 完成后进入哪里 |

原则：每个节点只承担一个学习动作，但节点内容必须是真实、完整、可照着练的 AI 使用过程。

## 13. 后台工作流

| 状态 | 行为 |
|---|---|
| 草稿 | 编辑内容，不在前台公开 |
| 预览 | 生成前台预览链接，供检查结构和内容 |
| 待审核 | 编辑提交给审核发布者 |
| 已发布 | 前台可访问，进入搜索和推荐 |
| 已下线 | 前台不可访问，但保留后台记录 |

发布前必须自动或人工检查：

1. 场景是否包含完整四阶段
2. 每个阶段是否包含能力标准、学习重点、练习任务、能力证据、AI 介入方式、工具建议
3. 每个已发布场景是否至少有一个可进入的学习节点
4. 节点是否包含 AI 对话示范和输出检查
5. slug、SEO、封面、排序是否完整

## 14. 工具与学习路径原则

AI之翼不做“最强工具排行榜”，工具模块必须服务学习推进。

| 原则 | 含义 |
|---|---|
| 工具服务学习阶段 | 不以工具为主语，以当前阶段学习目标为主语 |
| 能力标准先于工具建议 | 先明确要学会什么，再讨论用什么学 |
| 当前工具优先 | 先帮助用户把手头工具用到位 |
| 迁移有明确触发条件 | 只有当前工具无法支撑下一阶段，才建议迁移 |
| 工具建议后置 | 先看路径与标准，再看更优工具 |

## 15. 内容生产原则

| 原则 | 说明 |
|---|---|
| 所有内容必须挂载到场景与学习节点 | 不做孤立教程 |
| 每篇内容只承担一个节点职责 | 避免散乱 |
| 场景内容可持续扩展 | 后续新增场景只通过后台补内容 |
| 人定结构，AI 辅助扩写 | 核心资产是路径判断，不是文案产量 |
| 失败案例必须纳入 | 用户需要知道为什么学不会或做不稳 |
| 真实案例优先 | 展示工具选择、提示词、追问、补充、检查，而不是空泛说明 |

## 16. 非功能性需求

| 维度 | 要求 |
|---|---|
| 商业模式 | 免费运营网站，所有学习内容对用户免费开放 |
| 页面加载 | 首屏内容加载 ≤ 3 秒（4G 网络 / 家用宽带）；场景页和节点页核心内容加载 ≤ 2 秒 |
| 浏览器兼容 | 最近两个主版本的主流浏览器：Chrome、Edge、Safari、Firefox；不支持 IE |
| 响应式断点 | 桌面端 ≥ 1024px、平板 768–1023px、移动端 < 768px |
| 并发预期 | 首期支持 500 并发用户，架构上不设硬上限，可按需扩展 |
| 可用性 | 站点可用率 ≥ 99.5%（月度） |
| 无障碍访问 | 遵循 WCAG 2.1 AA 级基本要求：图片有 alt、表单有 label、键盘可导航、颜色对比度 ≥ 4.5:1、焦点可见 |
| SEO 基线 | 所有公开页面可被搜索引擎抓取；核心页面具备可配置 title、description 和结构化数据 |
| 安全基线 | HTTPS 全站；后台接口鉴权；用户数据加密存储；输入防注入；敏感操作有二次确认 |

## 17. 当前结论

AI之翼要建设的是一个成熟、可运营、可长期扩展的场景化 AI 学习网站。

交付完成后，前台用户能按场景学习，后台管理员能维护内容，后续新增场景只需要在后台补充场景、阶段、节点、案例和工具建议，不需要继续完善网站本身的设计和编程结构。
